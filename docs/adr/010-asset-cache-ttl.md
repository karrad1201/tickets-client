# ADR-010: Кеширование ассетов с TTL

**Статус:** `accepted`
**Дата:** 2026-04-09

## Контекст

Приложение загружает изображения (аватарки пользователей, обложки событий) по HTTP при каждом рендере. Это приводит к лишним сетевым запросам и мерцанию картинок при навигации назад/вперёд.

## Решение

Два объекта в пакете `data.cache`:

- **`ImageCache`** — in-memory хранилище `LinkedHashMap<String, Entry>` с TTL-проверкой:
  - `AVATAR_TTL_MS = 24h` — аватарки меняются редко.
  - `EVENT_IMAGE_TTL_MS = 1h` — обложки событий могут обновляться.
  - При `get()`: если запись просрочена — удаляется из store (lazy eviction).
  - Условие актуальности: `age < ttlMs` (строгое, чтобы TTL=0 всегда инвалидировал).

- **`CachedImageLoader`** — suspend-обёртка над Ktor `HttpClient`:
  1. Если URL пустой → null.
  2. Если в кеше есть актуальная запись → возвращает bytes из кеша.
  3. Иначе делает GET, кладёт bytes в кеш, возвращает bytes.
  4. При любой ошибке → null (UI рисует placeholder).

## Альтернативы

- **Coil / Kamel** — готовые image-loading библиотеки для Compose. Добавляют ~500 KB и требуют конфигурации под KMP. Избыточно для нашего числа изображений (~20 на экране).
- **Ktor `HttpCache` plugin** — кеширует на уровне HTTP (ETag/Last-Modified). Не даёт контроля над TTL и не позволяет явно инвалидировать по URL.
- **Disk cache** — переживает перезапуск, но требует `expect/actual` FileStore. Запланировано как следующий шаг (issue #63 комментарий).

## Последствия

- Изображения загружаются максимум раз в `ttlMs` вместо каждого рендера. ✓
- После обновления аватарки вызов `CachedImageLoader.invalidate(url)` инвалидирует кеш. ✓
- Кеш живёт в памяти процесса — после перезапуска пуст. Приемлемо для MVP. ✓
- Нет ограничения на размер кеша — контролируется вызывающим кодом (разумно при ~20 картинках в сессии). ✓

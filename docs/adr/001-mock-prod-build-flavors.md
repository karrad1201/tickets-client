# ADR-001: Две сборки через Gradle product flavors (mock / prod)

| Поле | Значение |
|---|---|
| Статус | Принято |
| Дата | 2026-04-08 |

## Контекст

Приложение работает с бекендом (Spring Boot). В процессе разработки бекенд может быть недоступен — нужна сборка, которая работает полностью автономно и демонстрирует весь функционал без сети.

## Решение

Два Android product flavor:

| Флейвор | `USE_MOCK` | applicationId | Назначение |
|---|---|---|---|
| `mock` | `true` | `…tickets.mock` | Разработка без сети, демо, PR-ревью |
| `prod` | `false` | `…tickets` | Продакшн-сборка с реальным API |

Флаг `BuildConfig.USE_MOCK` передаётся в `AppContainer.init(useMock)`, который выбирает между реальным и фейковым сервисом для каждого слоя.

### Контракт: каждый сервис реализуется дважды

```
XxxService (interface)
├── XxxApiService   — реальный Ktor-клиент  →  prod
└── FakeXxxService  — детерминированные хардкод-данные  →  mock
```

Оба реализуют один и тот же интерфейс — экраны не знают, какой именно сервис используется.

### Команды сборки

```shell
# Mock (без сети)
./gradlew :composeApp:assembleMockDebug

# Prod
./gradlew :composeApp:assembleProdDebug
./gradlew :composeApp:assembleProdRelease
```

## Альтернативы

- **Feature flags в рантайме** — требует UI для переключения, сложнее в поддержке.
- **Отдельный модуль :fake** — избыточно для текущего размера проекта.
- **Hardcode в AppSession** — не масштабируется, нет единого контракта.

## Последствия

- Каждый новый сервис **обязан** иметь `FakeXxxService`; PR без фейка не принимается.
- Фейковые данные должны покрывать все состояния: успех, ошибку, пустой список.
- Mock-сборка устанавливается рядом с prod (разные applicationId) — удобно для тестирования.

# Contributing

## Принципы

- Код пишется честно: понятно, поддерживаемо, без грязи под ковром
- Изменения идут через ветки и PR — не напрямую в `main`
- Неочевидные архитектурные решения фиксируются в [ADR](adr/)
- Разработка по TDD: Тесты → Код → Рефакторинг

---

## Git-культура

### Ветки

Формат имени ветки:
```
<тип>/<краткое-описание>
```

Типы:
- `feat/` — новая фича
- `fix/` — исправление бага
- `refactor/` — рефакторинг без изменения поведения
- `chore/` — зависимости, конфиги, скрипты
- `docs/` — только документация

Примеры:
```
feat/auth-phone
fix/event-card-overflow
refactor/ticket-viewmodel
```

### Коммиты

Формат сообщения ([Conventional Commits](https://www.conventionalcommits.org/)):
```
<тип>(<область>): <что сделано>
```

Примеры:
```
feat(auth): add phone number input screen
fix(event): fix price label alignment
refactor(home): extract event card to component
docs(adr): add ADR-001 navigation approach
```

Правила:
- Коммит — одна логическая единица изменений
- Не коммитить сломанный код
- Не смешивать рефакторинг и фичи в одном коммите

### Pull Request

- Каждый PR решает одну задачу
- Заголовок PR в том же формате, что и коммит
- Описание PR содержит: что сделано, почему, как проверить
- PR проходит ревью перед мержем в `main`
- Мерж только после одобрения

---

## Флоу добавления фичи

1. Создать ветку от актуального `main`
2. Написать тест (падает)
3. Написать минимальный код (тест проходит)
4. Рефакторинг
5. Если принято нетривиальное решение — создать ADR
6. Открыть PR, пройти ревью
7. Смержить

---

## ADR (Architectural Decision Records)

Фиксируем решения, которые:
- неочевидны и могут вызвать вопрос "почему так?"
- имеют альтернативы, которые были отброшены
- влияют на структуру или подходы в проекте

Создавать ADR: скопировать [шаблон](adr/000-template.md), присвоить следующий номер.

Индекс всех ADR: [adr/README.md](adr/README.md).

## Кодовый стиль

- Следуем [Kotlin Coding Conventions](https://kotlinlang.org/docs/coding-conventions.html)
- Compose-компоненты — в отдельных файлах, один файл — один экран или компонент
- `commonMain` — только платформо-независимый код
- Платформенный код — только там, где иначе нельзя
